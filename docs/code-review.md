# Code review — как описывать процесс на собесе

Это самый «мягкий» вопрос на собесе, но проваливают его именно потому что говорят расплывчато. Готовь конкретику.

---

## Как процесс шёл у нас (готовый рассказ)

> «Любые изменения вливаются только через MR в GitLab. Перед MR разработчик прогоняет локально build и test. В MR обязательно описание: что меняем, зачем, как тестировали, ссылка на Jira/задачу. На MR автоматически крутится pipeline: build, unit-тесты, integration-тесты на Testcontainers, SonarQube quality gate (coverage не ниже 70%, no new bugs/code smells выше Major).
> 
> Минимум 2 approvers, один обязательно — из core-команды сервиса. По нашему правилу — **не свой коммит, не своё ревью**. Если меняешь критичный модуль (платежи, миграции БД), нужен +1 от тимлида.
> 
> Reviewer обычно проверяет за час-два, на крупные MR — день. Если есть замечания — discussion в GitLab, требуется resolve каждой. Не разрешаем merge с unresolved discussions. После approve — squash merge, чтобы в master был один commit на фичу с осмысленным сообщением».

## Чек-лист, по которому ревьювил

### Корректность
- Логика делает что нужно по требованиям. Покрыты edge cases (null, пустой список, negative number).
- Транзакции: правильный scope `@Transactional`, понимает разработчик что внутри транзакции и где она кончается. Нет вызовов внешних API внутри транзакции (долгая транзакция = долгие локи).
- Race conditions: если два потока попадут одновременно — что будет? Optimistic/pessimistic locking, идемпотентность.

### API
- REST naming: существительные множественного числа, правильные HTTP методы, правильные статусы (201 на create, 204 на delete без тела, 409 на конфликт).
- Backward compatibility: новое поле не делает старых клиентов несовместимыми? Не убираем поле сразу.
- Error response — единый формат, есть `code` для машинной обработки, не отдаём stacktrace.

### БД
- Миграции backward-compatible? Не блокируют старый код?
- Индексы под query, EXPLAIN ANALYZE для подозрительных.
- N+1: `JOIN FETCH` или `@EntityGraph` для коллекций.
- `LAZY` всегда, `EAGER` только если осознано.

### Kafka
- Producer key проставлен, события одной сущности идут в одну партицию.
- Consumer идемпотентен (есть processed_events или эквивалент).
- Manual ack, не закомитили offset до обработки.
- В `@Transactional` consumer не пишет в Kafka сразу — outbox.

### Безопасность
- Никаких паролей/токенов в коде или логах.
- Логирование PII (ФИО, телефон, паспорт) — маскируем.
- SQL injection: только параметризованные запросы (JPA/JPQL/PreparedStatement).
- Authorization check на уровне controller или service.

### Тесты
- Unit на бизнес-логику (без Spring, быстро).
- Integration через Testcontainers на репозитории и Kafka consumer.
- Тесты покрывают happy path **и** error path.
- Никаких `Thread.sleep` в тестах — Awaitility или synchronous approach.

### Код
- Понятные имена. Метод делает одно. SRP.
- Нет закомментированного кода и `TODO` без ticket.
- Нет дублирования.
- Логи: уровень адекватный (не INFO на каждый запрос с PII), есть traceId.

### Производительность
- Не делаем N запросов в цикле, batch.
- Не грузим всю таблицу — pagination.
- Кэш где это даёт выигрыш, с TTL.

## Что сказать про «как давал и получал ревью»

> «Отдельно учился давать конструктивную обратную связь. Стараюсь:
> - писать **почему**, а не только **что**: не "переименуй", а "имя `data` ничего не говорит, читатель не понимает что это, лучше `paymentRequest`".
> - различать **must-fix** и **nit** (косметика). Помечаю явно: `nit:` — необязательно, `must:` — блокер.
> - не ревьювить эго: критика кода, не человека. Не "ты не подумал", а "тут не учтён случай X".
> - на свой код реагирую без обиды, прошу пояснить если не согласен, иду к обсуждению с автором голосом если в комментариях затягивается».

## Истории, которые работают

**1. Поймал N+1**
> «Ревьювил MR, видел `for (order in orders) { order.getCustomer().getName() }`. Спросил автора, прогнал EXPLAIN — на 10к заказов было 10к запросов. Заменили на `@EntityGraph` с `customer`, стало 1 запрос. Время ответа упало с 2с до 80мс».

**2. Поймал не-идемпотентный consumer**
> «На ревью консьюмера платежей увидел, что после ack списание уже произошло, а ack делается до записи в БД. Если consumer упадёт между ними — at-least-once дублирует списание. Предложил поменять порядок: транзакция БД (insert в processed_events + бизнес-операция), потом ack. Решило проблему дублей».

**3. Поймал баг в backward-compatible миграции**
> «Автор делал `ALTER TABLE ... ADD COLUMN ... NOT NULL DEFAULT ...`. На таблице 50М строк это блокирующая операция на час. Предложил разбить: ADD nullable → backfill batch-ами → SET NOT NULL. Это zero-downtime подход».

## Если спросят «а если автор не согласен»

> «Сначала пытаюсь объяснить, дать ссылку (документация, бенчмарк, прецедент в кодовой базе). Если всё равно несогласие — вытаскиваем третьего (тимлид, senior). На голосование не идём, идём к аргументам. Если вопрос вкуса (стиль) — уступаю автору. Если корректность или производительность — стою на своём, эскалирую».
